# 사내 점심 투표앱 PRD

## 1. 배경과 문제 정의

점심시간마다 "오늘 뭐 먹지?"를 정하는 데 매번 시간이 소모된다. 특히 여러 명이 함께 먹는 팀 단위에서는 메신저 단체방에서 "아무거나", "난 다 좋아" 같은 답변이 반복되고, 목소리 큰 몇 명의 의견으로 결정되거나 결정이 지연돼 식당이 붐비는 피크 타임에 걸리는 문제가 있다.

핵심 문제:
- **의사결정 지연**: 메뉴/식당을 정하는 대화가 산발적으로 흩어져 결론까지 오래 걸린다.
- **참여 불균형**: 소극적인 구성원의 선호가 반영되지 않는다.
- **기록 부재**: 지난주에 뭘 먹었는지 남지 않아 메뉴가 겹치거나 특정 식당만 반복된다.

이 앱은 **후보를 올리고, 정해진 시간까지 투표해, 결과를 자동 집계**하는 가벼운 사내 도구로 위 문제를 해결한다.

> 가정: "사내" 구성원 대상이며 외부 공개 서비스가 아니다. 조직 규모·팀 구조 등 구체 수치는 확인되지 않아 §11에 가정으로 명시한다.

## 2. 목표 및 성공 지표

**목표**
- 점심 메뉴/식당 결정을 **몇 분 내 마감되는 구조화된 투표**로 대체한다.
- 모든 참여자가 동등하게 한 표씩 행사하도록 한다.
- 과거 투표 이력을 남겨 메뉴 반복을 줄인다.

**성공 지표 (제안값 — 실제 목표치는 §11에서 확정 필요)**
- 투표 생성 후 **결과 확정까지 평균 소요 시간** ≤ 10분.
- 투표 참여율(초대 인원 대비 실제 투표 수) ≥ 70%.
- 주간 활성 사용자(WAU) 대비 **재사용률**(다음 주에도 최소 1회 투표 참여) ≥ 50%.
- 투표 생성부터 첫 투표까지 진입 장벽: 로그인 후 **3탭 이내**로 투표 생성 완료.

## 3. 타깃 사용자와 사용 시나리오

**타깃 사용자**
- **일반 구성원(투표자)**: 후보에 투표하고 결과를 확인하는 다수 사용자.
- **투표 개설자(호스트)**: 오늘의 점심 투표를 만들고 후보를 등록, 마감 시간을 정하는 사람. 별도 권한 없이 누구나 될 수 있다(가정).

**사용 시나리오**
1. **오늘 점심 정하기**: 오전 11시, 팀원 A가 앱에서 "우리 팀 점심" 투표를 만들고 후보 3~4개(김치찌개집, 파스타집, 샐러드)를 올린 뒤 마감을 11시 40분으로 설정한다. 팀원들이 각자 한 표씩 던지고, 마감 시 최다 득표 메뉴가 자동 확정된다.
2. **동점 상황**: 마감 시 두 후보가 동점이면 앱이 이를 알리고 호스트가 재투표 또는 무작위 추첨으로 결정한다.
3. **이력 참고**: 후보를 올리기 전, 지난주에 뭘 먹었는지 이력에서 확인해 중복을 피한다.

## 4. 기능 요구사항

각 항목에 우선순위: **MVP** / **다음** / **나중**

- **투표 생성** `MVP`: 제목, 후보 목록(텍스트), 마감 시간을 지정해 투표를 만든다.
- **후보 등록/추가** `MVP`: 개설 시 후보를 입력한다. (개설자 외 참여자의 후보 추가 허용 여부는 §11 미결)
- **투표하기** `MVP`: 후보 중 1개 선택(단일 선택). 마감 전까지 변경 가능.
- **실시간 결과 집계** `MVP`: 현재 득표 현황을 막대/카운트로 표시.
- **마감 및 결과 확정** `MVP`: 마감 시간 도달 시 자동으로 투표 마감, 최다 득표 후보를 결과로 표시.
- **동점 처리** `MVP`: 동점 시 명시적으로 표시하고 재투표/추첨 안내.
- **로그인/사용자 식별** `MVP`: 사내 구성원 식별(1인 1표 보장에 필요). 인증 방식은 §11 미결.
- **투표 이력 조회** `다음`: 과거 투표와 결과를 날짜순으로 조회.
- **복수 선택 투표** `다음`: 여러 후보에 동시 투표(찬성 표시형).
- **알림** `다음`: 투표 시작·마감 임박·결과 확정 시 푸시/메신저 알림.
- **식당 정보 연동** `나중`: 후보에 지도/영업시간/메뉴 링크 첨부.
- **자동 추첨/룰렛** `나중`: "아무거나"용 랜덤 결정 모드.
- **팀/그룹 관리** `나중`: 팀 단위로 참여 범위를 나눠 투표.
- **통계 대시보드** `나중`: 자주 먹은 메뉴, 참여율 추이 등 분석.

## 5. 데이터 모델

핵심 엔터티와 관계(개념 수준):

- **User(사용자)** — 사내 구성원. 식별자·이름.
- **Poll(투표)** — 제목, 마감 시간, 상태(진행중/마감), 개설자.
- **Candidate(후보)** — 투표에 속한 선택지(메뉴/식당명).
- **Vote(표)** — 한 사용자가 한 후보에 던진 표.

관계:
- User `1:N` Poll (한 사용자가 여러 투표를 개설)
- Poll `1:N` Candidate (한 투표에 여러 후보)
- Candidate `1:N` Vote (한 후보에 여러 표)
- User `1:N` Vote / Poll 기준으로는 User와 Poll이 Vote를 통해 `N:M` (한 사용자가 여러 투표에 참여, 한 투표에 여러 사용자 참여)
- **1인 1표 제약**: (User, Poll) 조합당 Vote는 최대 1개(단일 선택 기준). 복수 선택 도입 시(§4 다음) 이 제약은 (User, Poll, Candidate)로 완화된다.

> **물리 스키마(컬럼 타입·인덱스·마이그레이션) 확정은 범위 밖 — 개발 단계 결정.** 위는 개념 모델이며, 실제 테이블 설계·제약·인덱스는 구현 시 결정한다.

## 6. 사용자 흐름

**A. 투표 개설 흐름 (호스트)**
1. 로그인 → 홈(진행 중 투표 목록)
2. "투표 만들기" → 제목 입력 → 후보 추가 → 마감 시간 설정 → 생성
3. 생성된 투표 링크/카드를 팀에 공유

**B. 투표 참여 흐름 (투표자)**
1. 로그인 → 홈 또는 공유된 투표 진입
2. 후보 중 하나 선택 → 투표 → 현재 집계 확인
3. (마감 전) 필요 시 선택 변경

**C. 마감·결과 흐름**
1. 마감 시간 도달 → 투표 자동 마감
2. 최다 득표 후보를 결과로 표시 (동점 시 안내)
3. 결과가 이력에 저장

## 7. 화면 목록 및 화면별 요구사항

- **로그인 화면**: 사내 계정으로 로그인. 인증 실패 안내. (진입 표준 화면)
- **홈/대시보드**: 진행 중인 투표 목록(카드), "투표 만들기" 진입 버튼, 오늘의 확정 결과 요약. 진행 중 투표가 없을 때 빈 상태(empty state) 안내.
- **투표 생성 화면**: 제목 입력, 후보 추가/삭제(동적 리스트), 마감 시간 선택기. 유효성 검사(후보 2개 이상, 마감 미래 시각).
- **투표 상세/참여 화면**: 후보 리스트 + 선택 UI, 현재 득표 현황, 마감까지 남은 시간, 내 표 표시. 마감 후에는 결과 확정 표시로 전환.
- **결과 화면**: 최종 순위, 확정 후보 강조, 동점 시 재투표/추첨 액션.
- **이력 화면** `다음`: 지난 투표·결과 목록.
- **설정 화면**: 알림 on/off, 다크모드 토글 등 개인 설정.
- **프로필 화면**: 사용자 이름·부서 표시, 로그아웃.

> 셸 표준 화면(로그인/홈/설정/프로필)을 모두 포함한다. 이력 화면은 MVP에서 제외될 수 있으며(§4에서 `다음`), 그 경우 §10에 반영한다.

## 8. 디자인 방향

- **톤**: 가볍고 캐주얼하며 빠른 결정을 돕는 명료한 UI. 텍스트 최소화, 큰 터치 타깃, 한눈에 보이는 득표 막대. 점심이라는 일상적 맥락에 맞는 친근한(하지만 사내용으로 과하지 않은) 톤.
- **대상 뷰포트**: **모바일 우선(mobile-first)**. 점심 정하기는 대부분 폰에서 이뤄진다고 가정. 데스크톱은 반응형으로 대응(중앙 정렬 또는 full-width, 구현 시 결정).
- **다크모드**: **지원**. 시스템 설정 연동 + 설정 화면에서 수동 토글 제공.

## 9. 비기능 요구사항

- **성능**: 홈·투표 상세 초기 렌더 체감 지연 최소화. 득표 집계는 준실시간(폴링 또는 갱신 간격 확정은 구현 단계)으로 반영. 동시 투표 시에도 집계 정합성 유지.
- **보안/프라이버시**: 사내 구성원만 접근(인증 필수). 1인 1표를 서버에서 강제(클라이언트 신뢰 금지). 누가 무엇에 투표했는지 공개 범위는 §11 미결(익명 집계 vs 실명 표시).
- **접근성**: **WCAG 2.1 AA** 준수 목표 — 명도 대비, 키보드 조작, 스크린리더 레이블, 색상에만 의존하지 않는 득표 표시(수치 병기).
- **가용성**: 점심 시간대(오전 11시~오후 1시) 부하 집중 구간에 안정적 동작.

## 10. 범위 외 (Out of Scope)

- 실제 **음식 주문·결제·배달 연동** (앱은 "무엇을 먹을지" 결정까지만).
- **외부(사외) 사용자 공개**·회원가입 시스템.
- 식당 **평점·리뷰 시스템**.
- 급여·근태 등 **타 사내 시스템 통합**.
- (MVP 기준) 이력 조회·알림·팀 관리·통계 대시보드 — §4에서 `다음`/`나중`으로 분류된 항목은 MVP 범위 밖.
- 물리 DB 스키마·인프라 아키텍처 확정(개발 단계 결정).

## 11. 미해결 이슈와 가정

검증되지 않은 가정과 결정이 필요한 항목:

- **[가정] 인증 방식 미확정**: 사내 SSO/Google Workspace/사번 로그인 중 무엇을 쓸지 미정. 1인 1표와 사용자 식별에 직접 영향 → 확정 필요.
- **[가정] 조직 규모·이용 단위**: 전사 단일 투표인지, 팀/부서 단위인지 확인 안 됨. 팀 관리 기능(§4 나중)의 필요성과 MVP 범위에 영향.
- **[미결] 후보 추가 권한**: 개설자만 후보를 올리는지, 참여자도 추가 가능한지 미정.
- **[미결] 투표 공개 범위**: 누가 무엇에 투표했는지 실명 노출할지, 집계만 익명 공개할지 미정(프라이버시·참여 심리에 영향).
- **[미결] 동점·마감 정책**: 동점 시 자동 추첨 vs 호스트 수동 결정, 마감 후 재개설 허용 여부.
- **[미결] 성공 지표 목표치**: §2의 수치(참여율 70%, 소요시간 10분 등)는 제안값이며 실제 목표는 이해관계자 합의 필요.
- **[미결] 알림 채널**: 사내 메신저(Slack/Teams 등) 연동 가능 여부 미확인 → 알림 기능(§4 다음) 설계에 영향.
- **[가정] 실시간성 수준**: 득표 현황을 준실시간 폴링으로 가정. WebSocket 등 진짜 실시간이 필요한지는 사용 규모 확인 후 결정.
